# news-room 아키텍처 (논의 중)

> 전제: 전문 백엔드 없이, 갖고 있는 것(홈서버, Claude Code/Codex 기반 개인 에이전트 시스템)으로
> 단순하게 시작한다. 하루 1기사라는 특성상 서버가 상시 떠 있을 필요가 없다.

## 전체 그림

```
[홈서버 — 매일 정해진 시각 (cron/스케줄러)]
    │
    ├─ 1. 편집국 파이프라인 실행 (Claude Code / codex app-server 에이전트)
    │     ① 파수꾼: 한국 뉴스 지형 수집 (메이저 언론 + 커뮤니티·SNS)
    │     ② 등대: 오늘의 주제 선정, 객원 전문가 정의
    │     ③ 토론: 느티나무·물길·등에·청진기·씨줄·저울 + 객원
    │     ④ 등대: 토론 종합, 기사 작성
    │
    ├─ 2. 산출물을 레포에 저장 (마크다운)
    │     content/2026-06-11/
    │       article.md        ← 기사 본문
    │       debate.md         ← 토론 전문
    │       guest.md          ← 오늘의 객원 페르소나/프롬프트
    │       sources.md        ← 출처 목록 (article.md에 포함시킬 수도)
    │
    └─ 3. git commit & push
            │
            ▼
[GitHub] ── GitHub Actions: 정적 사이트 빌드 ──▶ GitHub Pages 배포
```

- 데이터베이스 없음. **git 레포 자체가 데이터베이스이자 아카이브.**
- 상시 실행 서버 없음. 하루 1회 배치만 돌면 됨.

## 이 구조의 숨은 장점: 프롬프트 공개가 저절로 된다

편집국 에이전트들(강령, 페르소나, 프롬프트)을 이 레포 안의 파일로 정의하면 —
레포를 public으로 두는 것만으로 "우리의 렌즈를 공개한다"(강령 6)가 구현된다.
독자는 사이트에서 보는 프롬프트가 실제로 기사를 만든 그 파일임을 git 히스토리로 검증할 수 있다.
강령·프롬프트의 수정 이력(강령 7의 개선 루프)까지 커밋 로그로 투명하게 남는다.

## 정해진 것

- 실행 환경: 홈서버의 스케줄러 (상시 백엔드 없음)
- 에이전트 런타임: **`../coco-agents` 워크플로 시스템** (Claude Code/Codex CLI 세션의 로컬 컨트롤 플레인)
- 정적 사이트 생성기: **Astro** (마크다운 콘텐츠 컬렉션이 일별 폴더 구조와 맞고, 토론 접기/펼치기 등 인터랙션이 가볍다. 개인 블로그 Hugo와 구분되는 선택)
- 배포: GitHub Pages 정적 사이트 (GitHub Actions 빌드)
- 저장: 레포 내 마크다운 (DB 없음)

## coco-agents 통합 방식

coco-agents의 워크플로 정의(JSON)는 에이전트별 `backend`(claude/codex)·`role`(프롬프트)·`budget`과
`task`/`fanin` 스텝, `needs` 의존성, 아티팩트 출력을 지원한다. 편집국 파이프라인을 그대로 표현 가능.

- **편집국 정의는 news-room 레포에 둔다**: 강령·페르소나는 마크다운으로, 파이프라인은
  `workflows/daily-newsroom.json`으로. 워크플로의 각 `role`은 해당 페르소나 마크다운을
  읽고 그 인물이 되라고 지시한다 (coco-agents의 기존 워크플로들이 쓰는 패턴).
  → 프롬프트 공개(강령 6)가 레포 공개로 저절로 구현되는 구조 유지.
- **실행은 coco-agents가 담당**: 홈서버 cron이
  `scripts/publish-daily.sh`를 실행하고, 래퍼는 `coco-agents session run`에
  `prompts/daily-newsroom-single-claude.md`의 내용을 전달한다. 이 프롬프트가
  `workflows/daily-newsroom.json`을 읽고 절차를 수행한다.
- 객원 전문가(매일 다른 페르소나)는 등대의 주제 선정 스텝이 `artifacts/guest.md`로 정의를
  출력하고, 객원 스텝의 role이 그 파일을 읽어 그 인물이 되는 방식으로 동적 처리.

### 워크플로 스케치

```
steps:
  1. 파수꾼(task)            → artifacts/news-landscape.md   (뉴스 지형 수집)
  2. 등대-주제선정(task, needs 1) → artifacts/topic.md + artifacts/guest.md
  3. 토론 1라운드: 느티나무·물길·등에·청진기·씨줄·저울·객원
     (task ×7, needs 2, 병렬)  → artifacts/round1-*.md       (각자의 시각)
  4. 토론 2라운드: 같은 멤버들 (needs 3 전체)                  (서로의 글을 읽고 반박·보완)
  5. 등대-기사작성(fanin, needs 4) → artifacts/article-draft.md
  6. 퇴고                         → artifacts/article.md
  7. 래퍼 조립                     → content/YYYY-MM-DD/{article,debate,guest,draft,prompt,run}.md
  8. git commit & push (래퍼 스크립트)
```

라운드 수(2라운드면 충분한가)는 구현하면서 실험으로 정한다.

### 세션과 워크스페이스의 관계

- **워크스페이스는 하나, 세션은 에이전트별.** coco-agents에서 워크플로 런의 모든 스텝은
  같은 실행 워크스페이스를 공유하고(`shared` 격리 모델), 각 스텝은 독립 세션(독립 컨텍스트)에
  바인딩된다. 비유: 사무실은 하나, 머리는 각자.
- 세션 간 소통은 아티팩트 파일로만 이루어진다 → 토론의 전 과정이 자연스럽게 파일로 남아
  "토론 전문 공개"(04 문서)가 부산물로 얻어진다.
- 공통 지침(워크스페이스의 CLAUDE.md/AGENTS.md)은 전 세션에 주입되고,
  페르소나 차별화는 각 스텝의 role(+ 페르소나 파일)이 담당한다.
- 같은 워크스페이스라 서로의 페르소나 파일을 읽을 수는 있다 — 막지 않는다
  (실제 편집국에서도 동료의 성향은 서로 안다). 단, 공통 지침에
  "토론 상대의 글에 반응하라, 상대의 페르소나를 흉내 내지 말라"를 명시한다.

### 워크스페이스 지침 문서 문제와 해법

coco-agents 세션은 특정 폴더(워크스페이스)에 바인딩되어 실행되므로, 그 폴더의
CLAUDE.md/AGENTS.md가 모든 에이전트 세션에 주입된다. 개발용 지침이 편집국 에이전트의
기사에 새어 들어갈 위험이 있다.

**해법: 오염 경로를 공통 시스템 프롬프트 주입구로 역이용한다.**

- 워크플로 실행 워크스페이스를 레포 내 전용 폴더(예: `newsroom/`)로 분리한다.
- 그 폴더의 CLAUDE.md·AGENTS.md는 **편집국 공통 지침**으로 작성한다
  (강령 요약, "너는 news-room 편집국의 일원이다", 산출물 규칙).
- 개발용 지침은 레포 루트에 두고, 편집국 워크스페이스에는 닿지 않게 한다.
- 이 공통 지침 문서 역시 공개된다 — 강령 6과 일관됨.
- 구현 시 스모크 런으로 에이전트에 실제 주입되는 컨텍스트를 검증한다.

## 실패 처리 ✅

**실패도 뉴스로 알린다.** 파이프라인이 실패한 날은 cron 래퍼(결정적 셸 스크립트)가
그날 자리에 "휴간 공지"를 커밋·발행한다 — 옛 신문의 결호 공지처럼. 실패를 숨기지 않는
것도 투명성의 일부.

- 래퍼 스크립트는 LLM과 무관한 셸이므로 파이프라인이 죽어도 거의 항상 동작한다.
- 래퍼마저 실패하는 경우(서버 다운, push 불가)는 사이트에 아무것도 안 올라온다 —
  편집자가 매일 기사를 읽는 루틴(강령 7)이 최후의 감지선. 필요해지면 그때 알림 추가.
